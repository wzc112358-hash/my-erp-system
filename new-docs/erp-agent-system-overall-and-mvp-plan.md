# ERP Agent 系统总计划与 MVP 分阶段方案

日期：2026-05-24

## 1. 背景与目标

当前 ERP 已经能记录销售、采购合同及相关业务信息，但它更像一个“信息登记系统”，还没有真正承担“发现商机、推动判断、沉淀决策依据、减少重复沟通”的职责。公司日常招投标信息仍依赖员工每天打开多个网站巡查，再在微信群里汇报“有无信息”和“是否相关”。这套方式能运行，但存在几个明显问题：

- 员工每天重复巡站，时间成本高。
- 群消息容易淹没，后续检索困难。
- “今天没有新增”“某网站失败”“某标错过截止期”等状态没有结构化记录。
- 标书、硬性要求、员工判断、王总确认过程分散在微信和个人经验中。
- 商机初筛依赖人工记忆，难以复盘为什么某个标当时被放弃或推进。

Agent 系统的第一目标不是替代员工或自动决定投标，而是把“发现、整理、提醒、初筛、分派、留痕”系统化。ERP 应成为正式处理入口，微信群只保留提醒习惯或过渡期摘要。

## 2. 总体定位

建议把公司 Agent 系统分成三个层次：

### 2.1 第一层：商机监测与报价闭环 Agent

这是最适合优先落地的方向。原因是输入明确、流程固定、收益可衡量。

核心能力：

- 自动巡检招投标网站。
- 识别新增公告。
- 提取标题、链接、发布时间、截止日期、采购单位、项目编号、附件信息。
- 按公司常做产品和化工助剂范围做初筛。
- 标记硬性投标条件：代理商限制、第三方检测、业绩、8 位码、危化品资质、生产商限制、截止期紧急程度。
- 分派给对应负责人确认。
- 形成王总确认包。
- 进入后续报价预审与合同机会管理。

### 2.2 第二层：采购与供应商 Agent

在商机确认“可关注”之后，采购 Agent 才有价值。它不应太早介入所有公告，否则会产生大量无效工作。

核心能力：

- 按产品规格寻找供应商。
- 对比历史采购价、供应商报价、交期、资质。
- 维护老供应商合作记录。
- 对进口供应商和外贸流程做资料清单、节点提醒、风险提示。

### 2.3 第三层：财务与经营分析 Agent

财务 Agent 最复杂，建议放到后续阶段。原因是财务数据准确性要求高，且涉及成本、流水、税务、报税、利润等高风险内容。

早期可先做辅助型能力：

- 合同毛利测算草稿。
- 应收应付提醒。
- 项目利润归集。
- 税票和回款状态看板。

不建议第一阶段就做自动报税、自动财务决策。

## 3. 总体架构

建议采用“中心 Agent 后端 + 站点采集工具 + ERP 交互界面 + 人工协助通道 + 本地采集助手”的架构。

```text
员工/王总
  |
  | ERP 页面：商机池、我的待判断、王总确认台、巡检记录、监测源管理
  v
PocketBase / ERP 数据层
  |
  | API / Webhook / Service token
  v
中心 Agent 后端
  |
  | 调度器、任务队列、去重、初筛、通知、审计日志
  |
  +-- HTTP/JSON/HTML 采集器
  +-- Playwright/CDP 浏览器采集器
  +-- 文档解析器：PDF、Word、Excel、图片 OCR
  +-- LLM 初筛与硬性条件抽取
  +-- 人工协助任务生成器
  +-- 本地采集助手接口
```

### 3.1 ERP 的职责

ERP 是正式业务入口，负责：

- 展示监测源配置。
- 展示每次巡检状态。
- 展示商机池。
- 让负责人确认“可关注、不相关、需王总判断、需补标书、已错过截止”。
- 让王总查看确认包并给出是否推进。
- 保存附件、截图、原始 HTML/JSON、LLM 提取结果和人工判断记录。

### 3.2 中心 Agent 后端的职责

中心 Agent 后端独立于 PocketBase 运行，负责：

- 定时巡检。
- 读取监测源配置。
- 选择对应站点采集策略。
- 生成巡检记录。
- 提取公告并去重。
- 触发详情页和附件解析。
- 调用 LLM 做初筛。
- 创建待确认任务。
- 生成群摘要或企业微信摘要。
- 记录失败原因和审计日志。

### 3.3 人工协助通道的职责

遇到登录、验证码、短信、CA、买标书、需要账号权限的详情页时，不应让服务器硬绕。第一阶段应让 Agent 在 ERP 生成任务：

- 需登录补详情
- 需上传标书
- 需人工确认截止日期
- 需人工复制公告正文
- 需人工截图

员工完成后，Agent 再继续解析和初筛。

### 3.4 本地采集助手的职责

本地采集助手是第二阶段或第三阶段能力，适合处理登录复杂、验证码频繁、服务器 IP 不适合访问的网站。

工作方式：

1. 员工电脑安装本地助手。
2. 助手登录 ERP，拉取该员工负责的采集任务。
3. 助手打开本机浏览器或连接员工本机 Chrome。
4. 员工手动登录、过验证码、完成必要点击。
5. 员工点击“继续采集”。
6. 助手抓取 DOM、网络响应、附件、截图并上传 ERP。

这样账号登录态留在员工本机，验证码由人处理，服务器负担也更低。

下面是根据现有文档和轻量探测得到的第一版判断。后续实现时，每个站点都应先做“侦察脚本”，确认真实接口、分页、详情页、附件规则，再进入稳定采集。

| 国能 E 招 | 小杨 | 服务端 HTML，列表、日期、链接清楚 | HTTP + HTML parser | 高 |

| 中化 SCM | 小杨/小魏文档涉及 | SPA，但公开 API 返回公告、采购人、发布时间、截止期 | API 优先，Playwright 补详情 | 高 |

| 隆道云 | 小魏 | 公开商机、分类、详情页可采，登录后更完整 | 公开先筛，命中后登录补详情 | 中 |

| 延长石油 | 小杨 | 文档和小冯链接中出现多个入口，需进一步确认主入口 | 先侦察公开公告页，再定策略 | 中 |
| 东华能源网 | 小杨 | 当前资料不足 | 补充入口后侦察 | 低 |
| 金能招标网 | 小魏文档 | 登录站点，相关性较少 | 低频巡检，人工补详情 | 低 |

## 5. 工具选型建议

### 5.1 生产采集底座

推荐主线：

- HTTP 客户端：用于静态 HTML、JSON API、OSS JSON。
- HTML parser：用于服务端页面列表和详情提取。
- Playwright：用于 SPA、需要点击、需要截图、需要捕获网络请求的站点。
- CDP 网络捕获：用于发现真实接口、保存请求和响应、辅助调试。

理由：

- 大部分公开公告不需要完整浏览器。
- 服务器配置低，应减少浏览器并发。
- Playwright 比 MCP 更适合作为生产服务里的浏览器控制库。

### 5.2 Scrapling

Scrapling 适合做 Python 采集 worker，尤其是：

- 页面结构经常变。
- 需要 adaptive selector。
- 需要一个从单页到 spider 的 Python 抓取框架。

不建议把它作为所有站点的唯一入口。对国能 E 招、国能 E 购、中化这类有清晰 HTML/JSON/API 的站点，用简单 HTTP/API 更稳。

### 5.3 Crawl4AI

Crawl4AI 适合：

- 把详情页转成 LLM 友好的 Markdown。
- 对长页面做抽取。
- 对附件和页面内容做研究型处理。
- 快速验证某个站是否适合通用抽取。

不建议在第一阶段把所有巡检都交给 Crawl4AI。商机监测需要确定性、可复盘、低成本，站点专用 parser 更可靠。

### 5.4 Chrome DevTools MCP

Chrome DevTools MCP 适合：

- 开发调试。
- 站点侦察。
- 查看网络请求。
- 手动验证页面是否渲染。

不建议作为生产定时服务。生产服务应使用 Playwright/CDP 代码，而不是依赖对话式 MCP 工具。

### 5.5 Browser Harness

Browser Harness 适合：

- 本地采集助手。
- 员工真实浏览器接管。
- 登录、验证码、文件上传下载、人机协同场景。
- 为特定网站沉淀 domain skill。

它不适合作为中心服务器上无人值守的第一选择，因为登录态、验证码、员工本机权限才是它的价值所在。

### 5.6 Firecrawl

Firecrawl 适合：

- 快速网页清洗。
- 通用搜索、抓取、Markdown 化。
- 监测和 change tracking 原型。

但招投标数据可能包含账号、标书、商业敏感信息。若使用云服务，需要谨慎评估数据合规。若自部署，对低配置服务器可能偏重。第一阶段不建议作为核心生产依赖。

## 6. 数据模型建议

已有或计划中的集合可以继续沿用：

- `monitor_sources`：监测源配置。
- `monitor_runs`：每次巡检记录。
- `bid_opportunities`：商机主表。
- `bid_documents`：标书、附件、截图、原始 HTML/JSON。
- `opportunity_reviews`：员工确认、王总判断、后续复核。
- `agent_audit_logs`：Agent 操作日志。

建议新增或扩展字段：

### 6.1 `monitor_sources`

- `source_name`：站点名称。
- `owner_user`：负责人。
- `strategy`：采集策略，如 `http_html`、`json_api`、`playwright_dom`、`playwright_network`、`manual_required`、`local_helper_required`。
- `entry_url`：入口地址。
- `category_paths`：需要巡检的栏目。
- `keywords`：关键词。
- `product_scope`：产品范围。
- `requires_login`：是否需要登录。
- `requires_captcha`：是否可能需要验证码。
- `credential_ref`：凭据引用，不存明文密码。
- `schedule`：巡检时间。
- `enabled`：是否启用。
- `failure_policy`：失败时是否提醒负责人。

### 6.2 `monitor_runs`

- `source`
- `started_at`
- `finished_at`
- `status`：`success`、`failed`、`partial_success`、`no_new`
- `new_count`
- `matched_count`
- `error_message`
- `artifact_refs`
- `agent_version`

注意：失败不能记为“无新增”。

### 6.3 `bid_opportunities`

- `source`
- `source_url`
- `source_unique_key`
- `title`
- `project_no`
- `buyer`
- `published_at`
- `deadline_at`
- `product_keywords`
- `raw_text`
- `status`
- `owner_user`
- `relevance_score`
- `risk_tags`
- `hard_requirements`
- `needs_manual_action`
- `manual_action_reason`

### 6.4 `opportunity_reviews`

- `opportunity`
- `reviewer`
- `role`：负责人、王总、采购、财务等。
- `decision`：可关注、不相关、需王总判断、需补标书、已错过截止、推进报价、放弃。
- `comment`
- `evidence_refs`
- `created_at`

## 7. 用户交互逻辑

### 7.1 员工视角

员工每天不再只是在群里写“今天无新增”，而是在 ERP 的“我的待判断”处理 Agent 推送的商机。

员工需要做：

- 查看自己负责网站的新增商机。
- 对每条商机确认状态。
- 对需要登录或标书的任务上传附件或补充正文。
- 对相关产品写初步判断。
- 对需要王总判断的商机提交确认包。

### 7.2 王总视角

王总不需要看所有原始公告，只看员工或 Agent 标记为“可关注/需判断”的商机确认包。

确认包包含：

- 公告标题、链接、采购单位、截止期。
- 产品名称、规格、技术参数。
- 标书或公告原文依据。
- 历史中标公示。
- 员工自评。
- Agent 风险点。
- 硬性条件标签。

### 7.3 微信群视角

第一阶段不建议直接自动操作普通个人微信群。

过渡方案：

- ERP 生成每日摘要。
- 员工一键复制到群。
- 如果后续迁移到企业微信，再接机器人/API 自动推送。

摘要只做提醒，不作为正式处理入口。

## 8. MVP 分阶段构建方案

### MVP 0：资料治理与采集源注册

目标：先把“人负责什么网站、网站怎么查、哪些需要登录、哪些栏目必须看”结构化。

范围：

- 整理 `monitor_sources`。
- 按群公告初始化负责人分工。
- 给每个站点标记采集策略和风险等级。
- 把账号信息从文档中移出，改为 `credential_ref`。
- 明确产品关键词和常做产品清单。

验收标准：

- ERP 能看到所有监测源。
- 每个站点都有负责人、入口、栏目、频率、登录/验证码标记。
- 没有明文密码进入代码仓库。

### MVP 1：公开站点自动巡检

目标：先让系统每天真的自动发现一批公开商机，并生成巡检记录。

优先站点：

- 国能 E 招
- 国能 E 购
- 中化 SCM
- 易派客公开公告
- 中海油公开公告
- 兵器/华锦公开列表

核心功能：

- 定时巡检：09:00、12:00、15:00、17:30。
- 成功、失败、无新增都生成 `monitor_runs`。
- 提取标题、链接、发布时间、截止期、采购单位、项目编号。
- 去重。
- 新商机进入 `bid_opportunities`。
- 初步关键词匹配。
- ERP 商机池展示。

验收标准：

- 每个启用站点每天有巡检记录。
- 同一公告不会重复创建。
- 失败会提醒负责人，不会显示为无新增。
- ERP 商机池和巡检记录一致。

### MVP 2：详情解析、附件上传与人工协助

目标：把“发现标题”升级为“能支持员工判断”。

核心功能：

- 抓取详情页正文。
- 下载或登记附件链接。
- 支持员工上传 PDF、Word、截图。
- 文档解析为文本。
- LLM 提取硬性条件：代理商、第三方检测、业绩、8 位码、危化品、生产商限制、截止期。
- ERP 生成“需人工补充”的任务。

验收标准：

- 每条相关商机都有详情文本或人工补充入口。
- LLM 标签必须带依据片段。
- 员工能在 ERP 内完成“可关注/不相关/需王总判断/需补标书/已错过截止”的确认。

### MVP 3：本地采集助手试点

目标：解决登录、验证码、买标书、服务器访问不稳定的问题。

试点站点：

- 云梦泽
- 中石油
- 兵器/华锦登录详情
- 隆道云登录详情
- 易派客登录详情

核心功能：

- 本地助手登录 ERP。
- 拉取员工自己的采集任务。
- 打开本机浏览器。
- 员工手动登录和验证。
- 助手采集 DOM、网络响应、附件、截图。
- 上传到 ERP。

验收标准：

- 至少 2 个登录站点可以通过“人登录，助手采集”的方式完成详情补充。
- 本地助手不保存明文密码。
- 上传内容能进入 `bid_documents` 并被后续解析。

### MVP 4：王总确认包与报价闭环

目标：把商机从“发现”推进到“是否报价”的决策闭环。

核心功能：

- 自动生成王总确认包。
- 员工补充规格、技术参数、历史中标公示、自评。
- 王总确认是否推进。
- 推进后创建报价预审任务。
- 后续接采购询价、成本测算、利润预估。

验收标准：

- 王总只看确认包，不需要翻微信群找上下文。
- 每个“推进/放弃”的决策都有依据。
- 可复盘某个商机从发现到决策的完整过程。

### MVP 5：采购、财务与经营 Agent 扩展

目标：在商机闭环稳定后扩展到更大范围的公司经营。

采购 Agent：

- 供应商推荐。
- 历史报价对比。
- 资质和交期风险。
- 进口流程清单。

财务 Agent：

- 合同毛利测算。
- 应收应付提醒。
- 税票状态提醒。
- 项目利润归集。

经营分析 Agent：

- 商机来源统计。
- 员工处理及时率。
- 产品机会热度。
- 报价成功率。
- 放弃原因分析。

## 9. 技术实施建议

### 9.1 中心 Agent 后端

建议使用当前已有 agent-service 继续演进，优先保持轻量：

- Node/TypeScript 负责调度、PocketBase API、Playwright、通知。
- Python worker 可选，负责 Scrapling、Crawl4AI、复杂文档解析。
- 第一阶段不引入复杂消息队列，可用数据库任务表加 cron。
- 后续任务量变大后再引入 Redis/BullMQ。

### 9.2 服务器资源策略

服务器配置低，因此：

- 默认 HTTP/API 采集。
- Playwright 并发限制为 1 或 2。
- 浏览器只在必要站点启动。
- 附件和截图设置保留周期。
- 大文件解析可以放到本地助手或异步 worker。
- 不在服务器上高频 5 分钟级巡检。

### 9.3 安全策略

- 不明文保存账号密码。
- 凭据使用环境变量、加密文件或专门的 credential store。
- 本地助手不上传 cookie，除非明确授权且加密保存。
- 所有 Agent 操作写审计日志。
- 对外部云抓取服务谨慎使用，尤其是标书、账号页、商业敏感内容。

## 10. 测试与验收计划

### 10.1 采集测试

- 每个站点保存 HTML/JSON fixture。
- parser 对 fixture 跑单元测试。
- 测试标题、链接、发布时间、截止期、采购单位提取。
- 测试分页。
- 测试空结果。
- 测试站点失败。

### 10.2 去重测试

- 同一 URL 不重复。
- 同一项目编号不重复。
- 同一标题和采购单位在短时间内不重复。
- 变更公告与原公告可关联但不覆盖。

### 10.3 初筛测试

- 常做产品能命中。
- 明显无关的工程、设备、服务类能降权。
- 需要人工判断的边界案例不直接丢弃。
- LLM 结论必须带依据。

### 10.4 业务流转测试

- 新商机进入负责人队列。
- 员工确认后状态正确变化。
- 需王总判断的商机进入王总确认台。
- 放弃、推进、需补资料都有记录。
- 巡检摘要和 ERP 商机池一致。

## 11. 主要风险与应对

### 11.1 网站变化风险

应对：

- 每个站点保留独立 adapter。
- 保存原始响应和截图。
- parser 有 fixture 测试。
- 失败后提醒负责人。

### 11.2 登录和验证码风险

应对：

- 不绕验证码。
- 生成人工协助任务。
- 后续用本地采集助手接管。

### 11.3 LLM 误判风险

应对：

- Agent 只初筛，不自动决定投标。
- 所有标签必须带依据。
- 低置信度进入人工判断。
- 员工最终确认状态。

### 11.4 数据安全风险

应对：

- 标书和账号页不走外部云服务。
- 凭据不入库明文。
- 附件访问按 ERP 权限控制。
- 所有下载和解析留审计日志。

### 11.5 员工使用习惯风险

应对：

- 第一阶段保留群摘要。
- ERP 页面只要求员工做最少确认动作。
- “无新增”由系统自动生成，直接减少员工负担。

## 12. 推荐第一步

下一步建议从 MVP 0 和 MVP 1 开始：

1. 确定监测源表字段和站点策略枚举。
2. 把群公告分工录入 `monitor_sources`。
3. 先做 3 个最稳站点：国能 E 招、国能 E 购、中化 SCM。
4. 每个站点完成：巡检、去重、入库、巡检记录、ERP 展示。
5. 再加入易派客和中海油。
6. 最后处理云梦泽、中石油、兵器登录详情、本地助手。

这个顺序能最快产生实际业务价值，同时避免一开始陷入验证码、登录、CA、买标书这些复杂问题。
